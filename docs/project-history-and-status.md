# 찾아봇 개발 히스토리 및 반영 현황

작성일: 2026-07-14

## 문서 목적

찾아봇 개발 과정에서 결정된 내용, 이미 반영된 기능, 아직 미완료인 작업, 보류 또는 기각한 방향을 한 곳에서 추적한다.

이 문서는 개발 중 잊으면 안 되는 기준 문서로 사용한다. 이후 기능을 붙일 때마다 아래 표의 상태를 갱신한다.

## 상태 기준

| 상태 | 의미 |
|---|---|
| 반영 완료 | 코드/데이터/배포 또는 검증까지 완료된 상태 |
| 반영 예정 | 방향이 확정되었고 개발 대기 중인 상태 |
| 확인 필요 | 계정, 키, URL, 정책 등 사용자 확인이 필요한 상태 |
| 보류 | 당장 하지 않지만 나중에 다시 볼 수 있는 상태 |
| 기각 | 현재 프로젝트 방향상 하지 않기로 한 상태 |

## 현재 큰 방향

찾아봇은 복지자원 추천 앱에서 출발하되, 범용 AI 챗봇 경험까지 열어둔다. 다만 교육 회차, 기관, 참가자 관리가 강한 흐름은 메인 앱에 직접 섞지 않고 교육용 베타 버전으로 별도 배포한다.

배포 방향은 두 갈래로 나눈다.

| 버전 | 목적 | 기본 UX | 상태 |
|---|---|---|---|
| 메인 버전 | 일반 사용자용 범용 찾아봇 | 복지 찾기, AI랑 대화, 사진 꾸미기 중심 | 반영 예정 |
| 교육 베타 | 현장 AI 교육/실습 관리 | 기관/회차/참가자/보관함/사용량 추적 중심 | 반영 예정 |

메인 버전은 간결한 범용 챗봇으로 유지하고, 교육 베타에서만 `광진노인종합복지관 교육 - 어르신`, `1회차`, `교육 날짜` 같은 맥락을 강하게 드러낸다.

운영 모드는 하나의 챗봇 안에서 다음처럼 나눈다.

| 모드 | 역할 | 상태 |
|---|---|---|
| 복지 찾기 | 기존 DB 기반 복지자원 추천, 전화/링크/찜/공유 | 반영 완료 |
| AI랑 대화 | 생일문자, 삼행시, 일상 대화, 쉬운 설명, AI 교육 실습 | 반영 예정 |
| 사진 꾸미기 | 사진 업로드 후 이미지 꾸미기, 이미지 생성/편집 | 반영 완료 |

순수 토글 방식만 사용하지 않는다. 사용자가 입력한 문장을 보고 자동 분기하되, 화면에는 현재 모드를 표시하고 필요하면 직접 전환할 수 있게 한다.

예시:

- "생활비가 빠듯해요" -> 복지 찾기
- "오늘 서울 날씨 어때? 복지관 가는데 옷차림 추천해줘" -> 날씨 안내 + 복지/외출 조언
- "생일 축하 문자 써줘" -> AI랑 대화
- "이 사진 예쁘게 꾸며줘" -> 사진 꾸미기

## 로그인 및 사용자 구분 정책

로그인은 다음 방식으로 진행한다.

| 입력값 | 사용 목적 | 저장 원칙 |
|---|---|---|
| 이름 | 현장 사용자 구분, 참가자 명단 매칭 | 원문 저장 최소화 |
| 전화번호 | 중복 사용자 구분, 참가자 명단 매칭 | 원문 저장 금지 또는 제한 |

구현 기준:

1. 모든 사용자는 첫 접속 시 이름과 전화번호를 입력한다.
2. 프로그램 참가자 명단과 일치하면 `participant`로 분류한다.
3. 명단에 없어도 현장 방문자는 `guest`로 입장 가능하게 한다.
4. 서버 내부에서는 `participantId` 또는 `userId`를 만들어 사용한다.
5. 전화번호 원문은 가능하면 저장하지 않고, 정규화 후 해시값으로 매칭한다.
6. 참가자 명단 엑셀은 `public` 폴더, 프론트 코드, GitHub에 올리지 않는다.
7. 개인정보 저장 안내는 별도 체크박스 대신 `계속하기` 버튼 아래 안내 문구로 제공한다.

권장 사용자 타입:

| 타입 | 조건 | 권한 |
|---|---|---|
| participant | 참가자 명단과 이름+전화번호 일치 | 전체 기능 사용 |
| guest | 명단에는 없지만 이름+전화번호 입력 | 현장 체험용 기능 사용 |
| admin | 관리자 별도 인증 | 로그/통계/평가 화면 접근 |

## 메인 버전과 교육 베타 분리 정책

교육 특화 온보딩은 메인 앱에 기본으로 넣지 않는다. 범용 챗봇으로 열어둔 상태에서 교육 회차 선택을 강하게 노출하면 일반 사용자가 왜 회차를 골라야 하는지 혼란스러울 수 있다.

따라서 다음 기준을 따른다.

| 항목 | 메인 버전 | 교육 베타 |
|---|---|---|
| 대상 | 일반 사용자, 현장 체험 사용자 | 교육 참가자, 운영진, 실습 현장 |
| 첫 화면 | 바로 대화/복지 찾기 중심 | 로그인 후 오늘 교육 시작 |
| 회차 선택 | 기본 노출 안 함 | 1~4회차/날짜 선택 |
| 사용자 구분 | 이름+전화번호 기반 간편 식별 | 참가자 명단 매칭 + 일반 체험 |
| 대화 보관함 | 사용자별 대화 보관 | 사용자+교육그룹+회차별 보관 |
| 이미지 업로드/꾸미기 | 기본 노출 안 함 | 버튼형 실습 기능으로 제공 |
| 사용량 제한 | 사용자/전체 기준 | 사용자/회차/교육일 기준 |
| 로그 분석 | 기능 사용, 질문 유형, 비용 | 교육 회차별 참여, 질문/답변, 실습 성과 |

권장 배포 방식:

| 방식 | 판단 | 이유 |
|---|---|---|
| 같은 배포 안에 `/education` 경로 추가 | 보류 | 구현은 쉽지만 일반 사용자와 교육 실험 설정이 섞일 수 있음 |
| 별도 Vercel 프로젝트 또는 별도 배포 도메인 | 권장 | 환경변수, 로그, GA4, DB 파티션, 사용량 제한을 분리하기 쉬움 |
| 완전 별도 repo | 보류 | 초기에는 중복 관리 부담이 큼 |

초기에는 같은 repo를 유지하되 `APP_VARIANT=main`과 `APP_VARIANT=education_beta`처럼 환경변수로 화면 흐름과 저장 정책을 나누는 것이 좋다. Vercel에서는 메인 배포와 교육 베타 배포를 별도 프로젝트 또는 별도 환경으로 관리한다.

OpenAI도 가능하면 프로젝트를 분리한다.

| OpenAI 프로젝트 | 사용처 | 이유 |
|---|---|---|
| `chajabot-main` | 메인 범용 챗봇 | 일반 사용량 추적 |
| `chajabot-education-beta` | 교육 베타 | 회차 실습 비용과 사용량 분리 |

## 교육 베타 이미지 기능 정책

이미지 업로드, 이미지 꾸미기, 이미지 생성은 전체 사용자에게 노출한다. 다만 비용 관리를 위해 기본 1일 3회 제한을 둔다.

현재 제공하는 이미지 기능:

| 기능 | 예시 | 입력 | 상태 |
|---|---|---|---|
| 이미지 생성 | "꽃다발 이미지 만들어줘" | 텍스트 | 반영 완료 |
| 사진 꾸미기 | "내 사진을 따뜻한 프로필 사진처럼 꾸며줘" | 업로드 이미지 + 텍스트 | 반영 완료 |
| 이모티콘 만들기 | "내 사진으로 귀여운 이모티콘 만들어줘" | 업로드 이미지 + 버튼 템플릿 | 반영 완료 |
| 카드/문구 이미지 | "생일 축하 카드 만들어줘" | 텍스트 + 선택 옵션 | 반영 완료 |

어르신 사용성을 위해 이미지 기능은 자유 입력만 열어두지 않고, 인기 요청을 버튼형으로 제공한다.

인기 요청 버튼 DB 예시:

| templateId | 버튼명 | 기능 | 기본 프롬프트 |
|---|---|---|---|
| `sticker_from_photo` | 내 사진으로 이모티콘 | image_edit | 업로드한 사진을 귀엽고 밝은 이모티콘 스타일로 꾸며줘 |
| `profile_warm` | 프로필 사진 꾸미기 | image_edit | 업로드한 사진을 자연스럽고 따뜻한 프로필 사진처럼 보정해줘 |
| `birthday_card` | 생일 축하 카드 | image_generate | 따뜻한 분위기의 생일 축하 카드 이미지를 만들어줘 |
| `flower_greeting` | 꽃 인사 이미지 | image_generate | 밝은 꽃과 짧은 인사 문구가 들어간 이미지를 만들어줘 |
| `family_message_card` | 가족에게 보낼 카드 | image_generate | 가족에게 보낼 다정한 안부 카드 이미지를 만들어줘 |

사용량 제한 기본값:

| 제한 항목 | 기본값 | 적용 위치 |
|---|---:|---|
| 사용자별 이미지 생성/편집 | 1일 3회 | 전체 사용자 |
| 회차별 이미지 총량 | 회차당 100회 | 교육 운영 시 별도 관리 필요 |

이미지 기능은 서버에서만 OpenAI API를 호출한다. 프론트엔드에는 API key를 절대 넣지 않는다.

## 교육 온보딩 및 회차 선택 정책

이 정책은 교육 베타 버전에만 적용한다.

교육 현장에서는 "누가", "어떤 교육 그룹에서", "몇 회차에", "어떤 대화를 했는지"가 중요하다. 따라서 로그인 이후 대화를 바로 시작하기보다 교육/체험 맥락을 최소한으로 선택하게 한다.

다만 어르신 사용성을 위해 첫 화면을 복잡한 탭 화면으로 만들지는 않는다. 가장 편한 구조는 다음과 같다.

1. 이름 + 전화번호로 로그인한다.
2. 참가자 명단과 매칭되면 교육 그룹을 자동 지정한다.
3. 오늘 진행 중인 회차가 있으면 "오늘 교육 시작하기"를 가장 크게 보여준다.
4. 다른 회차를 선택해야 할 때만 "다른 회차 선택"을 눌러 1~4회차 목록을 연다.
5. 명단에 없는 사용자는 "일반 체험 사용자"로 바로 들어갈 수 있게 한다.

추천 화면 흐름:

| 단계 | 화면 | 설명 |
|---:|---|---|
| 1 | 간편 로그인 | 이름, 전화번호, 자동 동의 안내 후 계속하기 |
| 2 | 시작 선택 | 오늘 교육 시작 / 다른 회차 선택 / 일반 체험 시작 |
| 3 | 대화 화면 | 선택된 그룹, 회차, 날짜를 상단에 작게 표시 |
| 4 | 새 대화/보관함 | 같은 회차 안에서 대화를 여러 개 생성·보관 |

회차 선택은 탭보다는 큰 카드 또는 리스트가 적합하다. 1~4회차가 항상 보이는 탭은 관리자는 편하지만, 첫 방문 사용자에게는 선택 부담이 될 수 있다. 따라서 기본은 "오늘 회차 바로 시작", 보조 기능으로 "다른 회차 선택"을 둔다.

교육 회차 데이터 예시:

| 필드 | 예시 | 설명 |
|---|---|---|
| groupId | `gwangjin-senior-education` | 교육/기관 그룹 ID |
| groupName | `광진노인종합복지관 교육 - 어르신` | 화면 표시용 그룹명 |
| userType | `participant` / `guest` | 참가자 또는 일반 체험 |
| sessionId | `round-1-2026-07-14` | 회차+날짜 단위 ID |
| sessionLabel | `1회차` | 화면 표시용 회차 |
| sessionDate | `2026-07-14` | 교육 날짜 |
| conversationId | `conv_...` | 개별 대화 ID |

현장 운영 최적화:

| 상황 | 권장 방식 |
|---|---|
| 정규 교육 참가자 | 로그인 후 명단 매칭, 오늘 회차 자동 제안 |
| 일반 체험 사용자 | 이름+전화번호 입력 후 `일반 체험 사용자`로 입장 |
| 회차별 QR 사용 가능 | QR URL에 `sessionId`를 넣어 회차 선택 생략 |
| 여러 회차 복습 | 보관함에서 회차별 대화 목록 확인 |

결론: 2-depth 구조는 필요하지만, 모든 사용자에게 무조건 깊게 보여주지 않는다. 로그인 후 "오늘 교육 시작"을 기본값으로 두고, 회차 변경은 보조 경로로 둔다.

## 대화 세션 및 보관함 정책

새 대화 시작과 대화 보관함은 반영한다.

필요한 이유:

1. AI 자유대화가 붙으면 대화가 길어질 수 있고, 전체 대화를 매번 모델에 보내면 비용과 지연이 증가한다.
2. 긴 대화는 모델 컨텍스트를 흐리게 만들어 복지 추천과 자유대화가 섞일 위험이 있다.
3. 현장 실습에서는 한 사람이 여러 주제로 실습할 수 있으므로 회차별 저장이 필요하다.
4. 사용자가 "아까 그 대화"를 다시 보고 싶을 수 있다.

구현 기준:

| 기능 | 반영 여부 | 구현 방향 |
|---|---|---|
| 새 대화 시작 | 반영 완료 | 보관함에서 빈 대화 시작 |
| 대화 보관함 | 반영 완료 | 날짜/첫 질문/모드 기준으로 대화 목록 표시 |
| 대화 이어하기 | 반영 완료 | 보관함에서 선택한 대화를 다시 열기 |
| 대화 삭제 | 반영 예정 | 사용자 요청 시 개별 회차 삭제 |
| 대화 제목 | 반영 완료 | 첫 질문을 짧게 줄여 자동 생성 |
| 컨텍스트 압축 | 일부 반영 | OpenAI 호출에는 최근 메시지만 전달. 세션 요약은 미완료 |

초기 구현은 브라우저 저장소로 시작할 수 있지만, 이름+전화번호 로그인을 도입하면 서버 DB 저장으로 확장하는 것이 좋다. 브라우저 저장소만 사용하면 기기 변경 시 대화 보관함이 이어지지 않는다.

PWA로 설치하더라도 대화 데이터가 자동으로 서버에 저장되는 것은 아니다. PWA는 앱 화면과 일부 파일을 기기에 설치/캐시하는 방식이며, 대화 내역 회수와 관리자 확인을 원하면 서버 DB에 저장해야 한다. 로컬 저장은 오프라인 임시 저장 또는 빠른 화면 복구용으로만 사용한다.

저장 기준:

| 저장 위치 | 역할 | 한계 |
|---|---|---|
| 서버 DB | 공식 대화 내역, 사용량, 회차별 통계 저장 | DB 선택과 개인정보 정책 필요 |
| 브라우저 IndexedDB/localStorage | 임시 캐시, 오프라인 대기열, 빠른 복구 | 기기 변경/캐시 삭제 시 사라짐 |
| GA4 | 클릭/모드/전환 이벤트 분석 | 원문 대화 저장 금지 |

OpenAI 호출 시 기본 정책:

| 항목 | 기준 |
|---|---|
| 자유대화 컨텍스트 | 최근 8~12개 메시지 + 세션 요약 |
| 복지 추천 컨텍스트 | 사용자 현재 질문 + DB 검색 결과만 사용 |
| 이미지/사진 컨텍스트 | 이미지 요청 회차만 별도 관리 |
| 오래된 대화 | 보관함에는 남기되 모델 호출에는 그대로 다 보내지 않음 |

## 추천 카드 UI 개선 정책

추천 자원 카드가 여러 개 나올 때 현재 구조는 새 요소가 추가될 때마다 대화 전체를 맨 아래로 스크롤한다. 이 때문에 사용자는 카드 묶음의 첫 번째 자원이나 안내 문구가 아니라 마지막 카드/후속 선택지를 먼저 보게 될 수 있다.

개선 방향:

1. 추천 결과가 도착하면 "추천 결과를 찾았어요" 안내 말풍선 또는 카드 묶음 시작점으로 스크롤한다.
2. 카드 3개가 한 번에 나올 때는 카드 묶음 상단을 먼저 보여준다.
3. 카드 내 자원명, 설명, 대상/지역/기간, 버튼 텍스트를 한 단계 키운다.
4. 상세 설명은 접기/펼치기를 유지하되, 기본 요약은 더 읽기 쉬운 줄간격으로 조정한다.
5. 카드 하단에 "다음 카드 보기" 같은 강제 안내를 넣기보다 자연스러운 스크롤 흐름을 유지한다.

구현 기준:

| 항목 | 현재 | 개선 |
|---|---|---|
| 카드 글씨 | 설명/메타 정보가 작음 | 자원명, 설명, 메타, 버튼 모두 소폭 확대 |
| 스크롤 | 항상 맨 아래 이동 | 추천 묶음 시작점으로 이동 |
| 답변 체감 | 카드가 지나가 버리는 느낌 | 안내 말풍선 + 카드 시작점 노출 |
| 접근성 | 작은 텍스트 부담 | 고령자 기준 가독성 우선 |

## 서버 DB 저장 설계

대화 내역, 이미지 요청, 사용량, 회차 정보를 회수하려면 서버 DB가 필요하다. PWA 로컬 저장은 보조 캐시로만 사용한다.

초기 DB 후보:

| 후보 | 장점 | 판단 |
|---|---|---|
| Supabase Postgres | SQL, Auth/Storage 확장, 대시보드 편함 | 권장 |
| Firebase Firestore | 실시간 동기화 편함 | 가능 |
| Vercel Postgres/KV | Vercel과 가까움 | 가능 |
| Google Sheets | 빠른 확인 쉬움 | 로그량/보안상 임시용만 |

권장 초기 선택은 Supabase다. 참가자, 회차, 대화, 사용량, 이미지 요청을 테이블로 분리하기 좋고, 나중에 관리자 화면을 만들기 쉽다.

권장 테이블:

| 테이블 | 저장 내용 |
|---|---|
| `users` | 내부 사용자 ID, 이름 해시/전화번호 해시, 사용자 타입 |
| `education_groups` | 기관/교육명, 운영 상태 |
| `education_sessions` | 회차, 날짜, 그룹 ID |
| `conversations` | 대화 ID, 사용자 ID, 회차 ID, 제목, 모드, 생성 시각 |
| `messages` | 대화별 user/assistant 메시지, 원문 저장 동의 여부 |
| `usage_events` | 기능별 호출 횟수, 토큰, 예상 비용, 차단 여부 |
| `image_jobs` | 이미지 생성/편집 요청, 템플릿 ID, 결과 상태 |
| `prompt_templates` | 인기 요청 버튼명, 기능 타입, 기본 프롬프트 |
| `resource_clicks` | 복지자원 카드 클릭, 전화 클릭, 공유 클릭 |

개인정보 저장 원칙:

1. 이름과 전화번호 원문은 가능하면 저장하지 않는다.
2. 로그인 매칭용으로 정규화 후 해시값을 저장한다.
3. 관리자 화면에 이름 표시가 꼭 필요하면 별도 동의 후 최소한으로 저장한다.
4. 대화 원문 저장은 명확한 동의 후 저장한다.
5. 이미지 업로드 원본은 오래 보관하지 않고, 처리 후 만료 정책을 둔다.

필수 환경변수:

| 환경변수 | 용도 |
|---|---|
| `OPENAI_API_KEY` | OpenAI API 호출 |
| `APP_VARIANT` | `main` 또는 `education_beta` |
| `DATABASE_URL` | 서버 DB 연결 |
| `APP_SECRET` | 해시/서명용 비밀값 |
| `GA4_MEASUREMENT_ID` | GA4 이벤트 |
| `KAKAO_JAVASCRIPT_KEY` | 카카오톡 공유 |

## AI 비용 및 제한 정책

OpenAI API를 새 프로젝트로 분리해 사용한다. 기존 다른 챗봇 프로젝트 API key는 재사용하지 않는다.

초기 예산은 30만원으로 가정한다. 앱 내부에서는 아래 제한을 기본값으로 둔다.

| 제한 항목 | 기본값 | 이유 |
|---|---:|---|
| 사용자별 자유대화 | 1일 80회 | 텍스트 대화는 저렴하므로 넉넉하게 허용 |
| 사용자별 이미지 생성 | 1일 3회 | 이미지 생성 비용 과다 방지 |
| 사용자별 사진 꾸미기 | 1일 2회 | 업로드 이미지 처리 비용 관리 |
| 전체 서비스 일일 예산 | 10,000원 | 평시 과금 폭주 방지 |
| 행사/실습일 일일 예산 | 30,000~50,000원 | 현장 실습일 임시 상향 |
| 월 전체 앱 한도 | 250,000원 | 30만원 중 예비비 5만원 확보 |

서버에서 기록할 항목:

| 항목 | 설명 |
|---|---|
| userId | 이름+전화번호 기반 내부 ID |
| userType | `participant`, `guest`, `admin` |
| date | 사용 날짜 |
| feature | `chat`, `vision`, `image_generate`, `image_edit`, `weather`, `welfare_search` |
| requestCount | 호출 횟수 |
| inputTokens | 입력 토큰 |
| outputTokens | 출력 토큰 |
| estimatedCostKrw | 예상 원화 비용 |
| providerUsage | OpenAI 응답의 usage 원본 |
| blocked | 한도 초과 차단 여부 |

프론트 localStorage만으로 사용량 제한을 걸지 않는다. 새로고침이나 브라우저 변경으로 우회될 수 있으므로 `/api/chat`, `/api/image` 같은 서버 API에서 호출 전 quota를 확인한다.

## DB 연결 전후 구분

지금까지 완료된 `이용 주체 선택`과 `이름+전화번호 로그인 화면`은 DB 없이도 동작하지만, 현재 운영 배포는 Supabase와 연결되어 있다.

현재 브라우저 저장값:

| 키 | 저장 내용 |
|---|---|
| `cb_subject` | 이용 주체 ID와 확정 시각 |
| `cb_user_profile` | 서버/로컬 userId, 이름 표시값, 마스킹 전화번호, 전화번호 뒤 4자리, userType |

전화번호 원문은 브라우저에 저장하지 않는다.

다음 작업부터 DB가 필요해지는 지점:

| 작업 | DB 필요 여부 | 이유 |
|---|---|---|
| 이름 + 전화번호 로그인 | 반영 완료 | Supabase `app_users` 저장 확인 완료 |
| 참가자 명단 매칭 | 반영 완료 | 테이블/API/UI/명단 임포트 확인 완료 |
| 대화 보관함 | 반영 완료 | 사용자별 대화 목록/이어보기/새 대화 시작 UI 반영 |
| 호출별 비용 카운팅 | 일부 반영 | `usage_events` 기록 반영. 원화 비용 계산 필요 |
| 주요 클릭 로그 | 반영 완료 | `usage_events.feature = event:*` 형태로 저장 |
| 교육 회차/날짜 선택 저장 | 반영 예정 | 교육 그룹, 회차, 사용자, 대화를 연결해야 함 |

사용자가 준비해야 할 후보 세팅:

| 선택지 | 사용자 준비물 | Codex가 할 일 |
|---|---|---|
| Supabase 권장 | Supabase 프로젝트 생성, DB URL 또는 API 키, 익명/서비스 키 구분 | 테이블 생성 SQL, 서버 API, Vercel 환경변수 연결 |
| Firebase/Firestore | Firebase 프로젝트, Web/API 설정, Admin 키 | 인증/DB 구조 설계 및 서버 연동 |
| Vercel Postgres/Neon | DB 생성, 연결 문자열 | SQL 테이블, 서버 API, 비용/로그 저장 |

현재 Supabase로 연결할 때 필요한 Vercel 환경변수:

| 환경변수 | 상태 | 비고 |
|---|---|---|
| `SUPABASE_URL` | 완료 | Supabase Project URL 또는 Data API URL. `/rest/v1` 포함 여부와 관계없이 서버에서 처리 |
| `SUPABASE_SECRET_KEY` | 완료 | 새 Supabase Secret key. 서버 전용. 절대 브라우저/공개 저장소에 넣지 않음 |
| `SUPABASE_SERVICE_ROLE_KEY` | 선택 | Legacy `service_role` key를 쓸 경우에만 사용 |
| `APP_SECRET` | 완료 | 이름+전화번호 해시용 긴 임의 문자열 |
| `ADMIN_TOKEN` | 필요 | `/admin.html` 운영 로그 조회/export용 관리자 토큰. Vercel에 추가해야 관리자 로그 화면 활성화 |
| `GA4_MEASUREMENT_ID` | 완료 | 기본값 `G-N4GEGDPSRJ`. Vercel 환경변수로 덮어쓸 수 있음 |

추가 환경변수로 조정 가능한 값:

| 환경변수 | 기본값 | 설명 |
|---|---|---|
| `OPENAI_CHAT_MODEL` | `gpt-4o-mini` | 자유대화 모델 |
| `OPENAI_CHAT_DAILY_LIMIT` | `80` | 사용자/IP별 1일 자유대화 임시 제한 |
| `OPENAI_CHAT_MAX_OUTPUT_TOKENS` | `700` | 자유대화 최대 출력 토큰 |
| `OPENAI_IMAGE_MODEL` | `gpt-image-2` | 일반 이미지 생성/편집 모델 |
| `OPENAI_STICKER_IMAGE_MODEL` | `gpt-image-2` | 내 사진으로 이모티콘 템플릿 전용 모델 |
| `OPENAI_IMAGE_QUALITY` | `medium` | 일반 이미지 품질 |
| `OPENAI_STICKER_IMAGE_QUALITY` | `medium` | 이모티콘 템플릿 품질. `high`는 결과 품질은 좋지만 16컷 편집에서 응답 시간이 길어질 수 있음 |
| `OPENAI_IMAGE_DAILY_LIMIT` | `3` | 사용자/IP별 1일 이미지 제한. Supabase 운영 환경에서는 `usage_events` 기준으로 계산 |

전화번호 전체 원문은 현재 구조에서 필요하지 않다. 참가자 명단 매칭은 `participants` 테이블의 참가자 ID와 전화번호 뒤 4자리/마스킹 값으로 충분하며, 전체 번호가 필요한 경우에도 SMS 인증이나 중복자 검증을 위한 일시 처리 후 해시화한다. 전체 번호가 포함된 원본 파일과 SQL은 `private/`에만 두고 GitHub에는 올리지 않는다.

## PWA 설치 정책

찾아봇은 네이티브 앱스토어 앱이 아니라 PWA로 설치한다. 사용자는 브라우저에서 직접 설치를 승인해야 하며, 서비스가 강제로 바탕화면에 설치되도록 만들 수는 없다.

반영된 항목:

| 항목 | 상태 | 메모 |
|---|---|---|
| 앱 아이콘 | 반영 완료 | 사용자가 제공한 찾아봇 파비콘을 192/512/180/32px로 변환 |
| Manifest | 반영 완료 | `public/manifest.webmanifest` |
| Service worker | 반영 완료 | 앱 shell, 주요 이미지, DB JSON 캐시. `/api/` 응답은 캐시하지 않음 |
| 설치 버튼 | 반영 완료 | 설정 화면의 `홈 화면 설치` 버튼 |
| PWA 이벤트 | 반영 완료 | `pwa_install_click`, `pwa_install_available`, `pwa_install_result`, `pwa_installed`, `pwa_sw_ready` |

PWA로 설치해도 대화 데이터가 기기에만 저장되는 것은 아니다. 현재 대화 원문과 주요 사용 기록은 Supabase DB 저장을 기준으로 하며, PWA 캐시는 화면과 정적 데이터의 빠른 실행을 돕는 보조 장치다.

GA4에서는 PWA와 일반 브라우저 접속을 같은 Web 데이터 스트림에서 본다. 별도 네이티브 앱 스트림은 만들지 않는다. 대신 이벤트에 `mode=browser`, `mode=standalone`, `mode=standalone_ios` 값을 붙여 설치 실행 여부를 구분한다.

## 로그 및 개인정보 원칙

질문/답변 로그는 서비스 개선에 중요하지만 개인정보가 섞일 수 있다.

따라서 다음 원칙을 따른다.

1. GA4에는 원문 질문/답변을 보내지 않는다.
2. GA4에는 모드, 버튼, 성공 여부, 비용 구간, 질문 유형만 보낸다.
3. 원문 질문/답변 저장은 별도 동의 후 서버 DB에 저장한다.
4. 이름, 전화번호, 생년월일, 건강 정보, 가족 정보 등은 최소 수집한다.
5. 로그 조회 화면은 관리자 인증 뒤에만 접근하게 한다.

| 네이버클라우드 우선 AI 연동 | 보류 | OpenAI API 결제 가능성이 확인되어 자유대화/이미지 기능은 OpenAI 우선 |
| 프론트엔드에 API key 직접 삽입 | 기각 | 보안상 금지 |
| GA4에 질문/답변 원문 저장 | 기각 | 개인정보 및 민감정보 유출 위험 |
| 완전 토글형 AI 모드만 제공 | 기각 | 어르신 사용자가 모드 선택에서 막힐 수 있음 |
| 기존 복지 추천 흐름에 자유 AI를 무분별하게 섞기 | 기각 | 복지 DB 근거성과 추천 품질 평가가 흐려짐 |
| 교육 회차 온보딩을 메인 앱 첫 화면에 강제 | 기각 | 범용 챗봇 사용자에게 교육 맥락이 과하게 노출되어 혼란 유발 |

## 확인 필요 목록

| 항목 | 확인자 | 메모 |
|---|---|---|
| OpenAI 새 프로젝트명 | 사용자 | 예: `chajabot-production` |
| OpenAI API key 발급 | 사용자 | 채팅에 키를 붙여넣지 말 것 |
| Vercel 환경변수 등록 권한 | 사용자 | `OPENAI_API_KEY` |
| 교육 베타 배포 방식 | 사용자 + Codex | 별도 Vercel 프로젝트 권장 |
| 앱 변형 환경변수 | Codex | `APP_VARIANT=main` / `APP_VARIANT=education_beta` 검토 |
| GA4 측정 ID | 완료 | `G-N4GEGDPSRJ` |
| Kakao JavaScript key | 사용자 | 카카오톡 직접 공유 시 필요 |
| 운영 도메인 | 사용자 + Codex | Vercel 배포 URL 재확인 |
| 화양복지관 공식 URL | 사용자 + Codex | 현재 도메인 DNS 실패 |
| 로그 저장 DB | 사용자 + Codex | Supabase/Firebase/Vercel DB 중 선택 |
