# 찾아봇 상황 발견형 질문 UX 설계

## 기획 의도

찾아봇은 특정 걱정만 묻는 상담형 챗봇이 아니라, 어르신이 복지서비스명을 몰라도 생활 맥락과 하고 싶은 활동을 말할 수 있게 돕는 복지자원 탐색 인터페이스다. 홈 화면의 첫 질문은 고정 문장이 아니라 랜덤으로 바뀌며, 질문 범위는 필요(생활형)와 욕구(여가·문화·배움)를 모두 포함한다.

## MVP 반영

- 홈 화면 첫 질문을 새로고침 또는 재진입 시 랜덤으로 노출한다.
- 빠른 질문을 `필요(생활형)`과 `욕구(여가·문화·배움)` 두 축으로 나눈다.
- `필요(생활형)`은 끼니, 돌봄, 건강, 생계, 안전, 주거 자원으로 연결한다.
- `욕구(여가·문화·배움)`는 경로당, 모임, 취미, 나들이, 문화행사, 디지털 배움 자원으로 연결한다.
- `질문 듣기` 버튼은 현재 홈 질문을 포함한 안내 문장을 음성으로 읽는다.
- 브라우저 자동재생 정책 때문에 음성 재생은 사용자 버튼 입력 이후 시작하도록 설계했다.

## 랜덤 첫 질문 후보

- 오늘은 무엇을 찾아드릴까요?
- 요즘 어떤 점이 걱정되세요?
- 지금 가장 필요한 게 있으세요?
- 요즘 해보고 싶은 게 있으세요?
- 어떤 걸 배우거나 즐기고 싶으세요?
- 혼자 지내시기 불편한 점이 있으세요?
- 오늘은 어떤 도움을 함께 찾아볼까요?

## 랜덤 입력 예시

- 예: 혼자 살아서 끼니가 걱정돼요
- 예: 스마트폰 쓰는 법을 배우고 싶어요
- 예: 경로당 모임에 나가고 싶어요
- 예: 병원 가기가 힘들어요
- 예: 생활비가 부족해요
- 예: 가까운 곳에서 즐길 거리가 있을까요?

## 두 축 분류

| 축 | 포함 범위 | 대표 UI 문장 |
|---|---|---|
| 필요(생활형) | 끼니, 돌봄, 건강·의료, 생계, 주거, 응급·안전 | 끼니·생활이 걱정돼요 / 건강이 걱정돼요 / 생활비가 빠듯해요 |
| 욕구(여가·문화·배움) | 경로당, 모임, 취미, 나들이, 문화행사, 디지털 배움 | 즐길 거리·모임을 찾고 싶어요 / 새로운 걸 배우고 싶어요 |

## 대화 첫 질문 후보

- 무엇을 도와드릴까요? 필요한 것도, 해보고 싶은 것도 편하게 골라주세요.
- 어떤 이야기든 좋아요. 어떤 점이 궁금하세요?
- 필요한 복지든 즐길 거리든, 어떤 걸 찾아드릴까요?

## 여가·문화 분기

`즐길 거리·모임을 찾고 싶어요`를 선택하면 `어떤 활동을 찾고 계세요?`라고 묻고, 다음 세 갈래로 이어진다.

- 경로당·모임에 나가고 싶어요
- 취미·여가 프로그램을 배우고 싶어요
- 나들이·문화 행사에 가보고 싶어요

## 대화 흐름

1. 홈에서 랜덤 첫 질문, 랜덤 입력 예시, 두 축 빠른 질문을 보여준다.
2. 사용자는 음성, 텍스트, 빠른 질문 중 편한 방식으로 상황을 표현한다.
3. 찾아봇은 질문을 축과 세부 카테고리로 분류한다.
4. DB 검색 결과를 근거 카드로 제시한다.
5. 사용자는 필요한 자원을 찜에 저장하고, 설정 화면에서 보호자·활동가 공유를 실행할 수 있다.
6. 응급·안전 표현이 감지되면 일반 추천보다 안전 안내와 공식 자원을 우선한다.

## 음성 기술 선택

### 현재 MVP

- STT: 브라우저 Web Speech API
- TTS: 브라우저 `speechSynthesis`
- 장점: 비용 없이 즉시 배포 가능
- 한계: 브라우저/기기별 음성 품질과 지원 범위 차이

### 운영 버전

- STT: NAVER Cloud CLOVA Speech
- TTS: NAVER Cloud CLOVA Voice
- API 키는 프론트엔드에 넣지 않고 Vercel Serverless Function 또는 별도 백엔드에 보관한다.
- 한국어 음성 품질과 운영 안정성을 고려하면 실증용 운영 스택으로 적합하다.

## 결과보고서 표현

> 찾아봇의 발견 방식은 두 축이다. 하나는 끼니·돌봄·건강·생계·안전처럼 결핍을 해결하는 필요(생활형)이고, 다른 하나는 모임·취미·나들이·디지털 배움처럼 삶을 넓히는 욕구(여가·문화·배움)이다. 기존 복지 포털은 사용자가 정확한 서비스명을 알아야 검색할 수 있지만, 찾아봇은 매번 다른 질문을 먼저 던져 두 축을 번갈아 노출함으로써 사용자가 대화 중에 받을 수 있는 자원을 발견하도록 설계하였다. 즉 검색이 아니라 대화가 발견의 경로라는 점이 핵심 차별점이다.
